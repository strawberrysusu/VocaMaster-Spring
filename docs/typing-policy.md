# Typing 모드 채점 정책 (ADR-026 결정 B)

> 사용자/면접 답변용 명세. 코드를 보지 않아도 정책을 한눈에 알 수 있도록 정리.

## 1. 목적

타이핑 채점은 *너무 엄격하면* 사용자가 화나고 *너무 관대하면* 학습 의미가 사라진다. 그 사이 **중간 길**을 명시적으로 정의하고 모두 같은 기준으로 채점.

## 2. 현재 정책 (정규화 3단계)

사용자 입력과 정답을 다음 순서로 처리한 뒤 비교한다.

### ① `trim()` — 양 끝 공백 제거
```
"  사과  "  →  "사과"
"사과 "     →  "사과"
```
단어 *중간* 공백은 유지 (예: `"ice cream"` → `"ice cream"`).

### ② `equalsIgnoreCase()` — 대소문자 무시
```
"Apple" vs "apple"  →  정답
"APPLE" vs "apple"  →  정답
```

### ③ 쉼표 복수 정답 분리 (Quizlet 패턴)
정답 필드에 `,`로 구분된 후보 여러 개를 둘 수 있다. **하나라도 일치하면 정답.**
```
correct_answer = "사과, 능금"
사용자 "사과"  →  정답
사용자 "능금"  →  정답
사용자 "apple" →  오답
```

각 후보는 다시 `trim()` + `equalsIgnoreCase()` 적용 후 비교.

## 4. 한계 (의도적 트레이드오프)

### 답 자체에 `,`가 포함되면 깨짐
정답 `"Hello, World"`는 `["Hello", "World"]`로 분리되어 둘 중 하나만 맞아도 정답 처리됨 — **현재 의도와 다른 결과**. 단어 단위 학습이 주된 사용처라 *드문 케이스*로 판단하고 미해결.

### 정규화는 *최소한*만
구두점(`.`, `?`, `!`), 한글 자모 분리(NFC/NFD), 띄어쓰기 오차 등은 *건드리지 않음*. 사용자가 *정확히 외운 것*을 확인하는 게 학습 가치이므로 과도한 관대는 피한다.

### 과거 attempt 결과는 재채점 안 함
정책이 변경되어도 이미 저장된 `typing_questions.is_correct`는 그대로. 사후 변경 시 *과거 통계와 일관성 깨짐* 방지.

## 5. STRETCH — Levenshtein (오타 허용)

| 정책 | 채택 여부 | 사유 |
|---|---|---|
| 1~2 글자 오타 허용 (Levenshtein 거리) | ❌ STRETCH | 정확 암기가 학습의 핵심. 오타 허용은 *틀린 철자*를 강화. 시간 남으면 *옵션 토글*로 도입 검토 |
