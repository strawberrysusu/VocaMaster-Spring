# 복습 알고리즘 — Leitner Box (ADR-029)

> 목표: 에빙하우스 망각곡선 대응 — **"모르는 단어는 자주, 아는 단어는 가끔"**.
> 코드: `ReviewService.recordAnswer` · `CardProgress` 엔티티 · `V7__add_card_progress.sql`

## 박스 간격표

| box | 간격 | 이 박스에 있다는 뜻 |
|---|---|---|
| 1 | 10분 | 방금 틀렸거나 처음 만난 카드 — 곧바로 재도전 |
| 2 | 1일 | 한 번 맞힘 — 하루 뒤에도 기억나는지 확인 |
| 3 | 3일 | 자리 잡기 시작 |
| 4 | 7일 | 단기 기억 → 장기 기억 이동 구간 |
| 5 | 14일 | 거의 외운 카드 |
| 6 | 30일 | 완성 — 한 달에 한 번만 점검 (최고 등급, 천장) |

간격이 점점 벌어지는 이유: 복습 직후에는 천천히 잊히므로, **잊기 직전쯤 다시 보여주는 것**이 같은 노력으로 가장 오래 기억하는 방법 (간격 반복의 핵심 아이디어).

## 규칙 4개

1. **맞힘** → `box + 1`, 천장 6 (`Math.min(box + 1, MAX_BOX)`).
   천장이 없으면 box 7이 되어 간격 배열(6칸, 인덱스 0~5) 밖을 읽고 `ArrayIndexOutOfBoundsException` → 500.
2. **틀림** → `box = 1` 풀 리셋 + `correctStreak = 0`. 단 `wrongCount`는 +1 누적 (이력은 지우지 않음 — 오답 통계 재료).
3. **처음 만난 카드** → 첫 답변 순간에 progress를 **lazy 생성** (box 1). 카드 생성 시 미리 만들지 않음 — 안 배우는 카드에 쓸데없는 행을 만들지 않고, Card 생성/삭제 코드를 건드릴 필요도 없음.
4. **날짜 도장** → 박스 이동이 끝난 **새** 박스의 간격으로 `nextReviewAt = now + 간격`.
   순서를 바꾸면(날짜 먼저) 옛 박스의 간격이 박제되는 버그 — 예외가 안 터지는 "조용한 버그"라 테스트로만 잡힌다.

## 왜 틀리면 무조건 box 1인가 (한 칸 강등이 아니라)

- 틀렸다 = "이 단어 못 외웠다"는 신호. box 5→4 강등이면 모르는 단어를 **7일이나 방치**하게 됨.
- 풀 리셋이면 10분 뒤 바로 재도전 — "모를수록 자주"라는 서비스 목적에 정확히 부합. 전통 Leitner 방식.
- **트레이드오프**: 30일 만에 만난 box 6 카드를 실수 한 번에 바닥으로 — 가혹함. 정확 암기 우선이라 의도적 선택이고, 유저 불만이 생기면 "한 칸 강등 모드" 옵션화 여지 있음 (= "다음 개선안?" 면접 질문의 답).

## 왜 Leitner인가 (vs SM-2 / FSRS) — 면접 30초 버전

> "간격 반복 알고리즘으로 SM-2(Anki 고전)와 FSRS(최신, ML 기반)도 검토했지만 Leitner Box를 선택했습니다.
> 첫째, **설명·통제 가능한 복잡도**가 우선이었습니다 — Leitner는 if/else 수준이지만 망각곡선의 핵심(맞으면 간격 늘리고 틀리면 줄인다)은 전부 담습니다.
> 둘째, FSRS는 사용자 복습 기록으로 파라미터를 학습하는데 **신규 서비스에는 그 데이터가 0**입니다 (cold start).
> 대신 복습 기록이 쌓이면 더 정교한 알고리즘으로 전환할 수 있게 진행 데이터를 남기고 있습니다."

상세 대안 비교는 ADR-029 (`docs/decisions.md`) 참조.

## 구현 매핑 — `recordAnswer` 6단계

| 단계 | 하는 일 | 실패 시 |
|---|---|---|
| ① | 카드 실존 확인 | 404 NotFound |
| ② | 소유권 검문 (`deckService.verifyOwner`) | 403 Forbidden — 없으면 로그인만 한 공격자가 남의 cardId를 찍어 조작 가능 (**IDOR**) |
| ③ | progress 조회, 없으면 box 1로 생성 | — |
| ④ | 박스 증감 (규칙 1·2) | — |
| ⑤ | **새 박스** 간격으로 nextReviewAt 계산 (규칙 4) | — |
| ⑥ | 저장 — 신규 progress는 `save()` 필수 (`new`로 만든 객체는 영속성 컨텍스트 밖이라 더티체킹 대상이 아님 → save 없으면 INSERT가 안 나가고 **200 응답인데 DB엔 없는** 최악의 조용한 버그) | — |

## `CardProgress` 필드 의미

| 필드 | 의미 | 성격 |
|---|---|---|
| `boxLevel` | 지금 몇 번 상자인가 (1~6) | 현재 상태 (오르내림) |
| `correctStreak` | 이 카드 연속 정답 **횟수** (일수 아님 — 연속 학습일 streak은 별도 개념) | 현재 상태 |
| `wrongCount` | 지금까지 총 몇 번 틀렸나 | 누적 이력 (리셋 없음) |
| `nextReviewAt` | 이 시각 지나면 복습 대상(due) | due 판정 기준 |
| `lastReviewedAt` | 마지막으로 답한 시각 | 이력 |
| `@Version` | 동시 답변 충돌 감지 (낙관적 락) | 인프라 |

## 면접 예상 질문 5

1. **왜 SM-2/FSRS 안 쓰고 Leitner?** → 위 30초 버전. 핵심 단어: 설명 가능한 복잡도, cold start, 개선 여지.
2. **틀리면 왜 완전 리셋? 너무 가혹하지 않나?** → 모를수록 자주. 강등이면 모르는 단어 7일 방치. 가혹함은 인지한 트레이드오프고 옵션화 여지.
3. **간격(10분~30일)의 근거는?** → 경험칙이지 논문 최적값 아님 (정직하게). 상수 배열 한 곳에서 관리해 조정 비용이 낮음.
4. **동시에 두 번 답하면?** → `@Version` 낙관적 락으로 충돌 감지. 충돌 시 409 or 1회 재시도 정책 (Phase 3 후반 구현 예정).
5. **correct 값을 클라이언트가 보내는데 조작 걱정은 없나?** → 복습 답변은 **자기평가**(플래시카드 "안다/모른다", Anki 방식)라 correct가 클라이언트 입력인 것이 설계 의도 — 속여봤자 자기 복습 큐만 망가짐. 반면 Quiz/Typing처럼 **정답이 존재하는 모드는 서버가 채점**하며, 향후 그 채점 결과로 `recordAnswer`를 호출해 연동 (후속 작업). 서버가 강제하는 것은 소유권(`verifyOwner`, IDOR 차단)과 박스 규칙 자체.

## 복습 대상(due) 조회

- due 판정: `nextReviewAt <= now`
- **새 카드(progress 없음)는 due에 포함하지 않는다** (2026-07-22 결정)
  - "복습(review)"의 말뜻 그대로 — *배운 것을* 다시 보는 것. 신규 학습은 기존 3개 학습 모드(플래시카드/퀴즈/타이핑) 담당
  - 새 카드가 복습 큐에 들어오는 입구 = 학습 중 첫 답변(`recordAnswer`)이 progress를 lazy 생성하는 순간
  - 대안이었던 "신규+복습 혼합(Anki 스타일)"은 학습 모드와 역할이 겹치고 쿼리만 복잡해져 배제. "새 카드 N개" 카운트 노출은 today-summary(SHOULD) 때 확장 여지
